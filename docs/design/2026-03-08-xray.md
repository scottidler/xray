# Design Document: xray — Context-Aware Codebase Explorer for Agents

**Author:** Scott Idler + Claude
**Date:** 2026-03-08
**Status:** In Review
**Review Passes Completed:** 5/5

## Summary

xray is a Rust CLI tool that lets agentic LLMs explore codebases efficiently by querying at different detail levels with enforced output budgets. It replaces wasteful exploration patterns (running `tree`, reading entire files) with targeted, budget-aware queries that minimize context window consumption while maximizing understanding.

## Problem Statement

### Background

Agentic LLMs need to understand codebases to perform software engineering tasks. They have finite context windows (typically 100K-200K tokens) that must hold instructions, conversation history, tool results, *and* codebase context simultaneously. Every line of exploration output competes with working memory.

Current exploration tools were built for humans, not agents:
- `tree` dumps entire directory structures including noise (`node_modules/`, `data/`, `examples/`, `__pycache__/`)
- `cat`/`Read` on individual files requires knowing what to read and loads full file contents
- `find`/`Glob` returns flat file lists with no structural context
- No existing tool lets an agent *budget* its exploration or progressively drill deeper

**Prior art:** `pyr` (at `~/repos/scottidler/pyr`) solved Python-specific code extraction well — parallel parsing with `rustpython-parser`, cascading pattern matching, YAML/JSON output with TTY detection, deterministic output via `BTreeMap`. However, it lacked:
- Multi-language support (Python only)
- Budget awareness (no way to limit output)
- Smart tree collapsing (no skeleton view at all)
- File categorization (no concept of source vs config vs test)
- Config-driven behavior (hardcoded ignore patterns)

### Problem

Agents waste 60-80% of their exploration budget on low-signal output. They read entire files when they need one function signature. They enumerate every file in `examples/` when a count would suffice. There is no tool designed from the ground up for agent-efficient codebase exploration.

### Goals

- Provide layered detail levels: cheap orientation (`skeleton`) and targeted extraction (`outline`)
- Enforce output budgets with actionable error messages when exceeded
- Auto-detect and support multiple languages (Rust, Python, TypeScript/JavaScript)
- Collapse low-signal directories to summary lines instead of full enumeration
- Drive all classification (hidden/collapse/expand, file kinds) from a global config file
- Produce structured output (YAML on TTY, JSON when piped) with a `lines:` footer on every response
- Enable composable filtering by file kind (source, config, test, ci, docs, build) and glob patterns

### Non-Goals

- **Not an IDE/LSP replacement** — xray extracts static structure, not live diagnostics
- **Not a code search tool** — use `grep`/`ripgrep` for content search; xray finds structure
- **Not a dependency auditor** — future `deps` layer may exist but is out of v1 scope
- **Not a diff tool** — future `diff` layer may exist but is out of v1 scope
- **No incremental/cached analysis in v1** — every invocation is a fresh scan
- **No remote repository support** — operates on local filesystem only

## Proposed Solution

### Overview

xray is a Rust CLI built with clap that accepts a target directory (defaulting to CWD) and a detail layer (`skeleton` or `outline`). Filters narrow the output by file kind, language, or glob pattern. A `--budget` flag enforces a maximum line count, returning an error with the overage amount if exceeded. Output format auto-detects based on TTY attachment (YAML for terminals, JSON for pipes) with `--format` override.

### Architecture

```
                    CLI (clap)
                       │
                ┌──────┴──────┐
                │   Dispatch   │
                └──────┬──────┘
                       │
          ┌────────────┼────────────┐
          │            │            │
     ┌────┴────┐  ┌────┴────┐  ┌───┴───┐
     │ Config  │  │ Detect  │  │ Walk  │
     │ Loading │  │  Lang   │  │ Files │
     └────┬────┘  └────┬────┘  └───┬───┘
          │            │           │
          └────────────┼───────────┘
                       │
               ┌───────┴───────┐
               │   Classify    │
               │ (file → kind) │
               └───────┬───────┘
                       │
              ┌────────┴────────┐
              │                 │
        ┌─────┴─────┐   ┌──────┴──────┐
        │  Skeleton  │   │   Outline   │
        │  (tree +   │   │ (parse +    │
        │  collapse) │   │  extract)   │
        └─────┬─────┘   └──────┬──────┘
              │                │
              └────────┬───────┘
                       │
               ┌───────┴───────┐
               │    Budget     │
               │   Enforce     │
               └───────┬───────┘
                       │
               ┌───────┴───────┐
               │    Output     │
               │  (YAML/JSON)  │
               └───────────────┘
```

**Data flow:**
1. Parse CLI args, load config (global `~/.config/xray/xray.yml` with fallback to embedded defaults)
2. Detect languages present in target directory via marker files
3. Walk filesystem using `ignore` crate (respects `.gitignore`)
4. Classify each file into a kind (source, config, test, ci, docs, build) using language-specific glob patterns from config
5. Execute the requested layer:
   - **skeleton**: Build directory tree, apply hidden/collapse/expand rules
   - **outline**: Parse source files with language-specific parsers, extract symbols
6. Count output lines against budget; error if exceeded
7. Serialize to YAML or JSON based on TTY detection or `--format` flag

### Data Model

#### Config Structure

```rust
struct Config {
    defaults: Defaults,
    directories: DirectoryRules,
    languages: HashMap<String, LanguageConfig>,
}

struct Defaults {
    budget: usize,       // 0 = unlimited
    format: FormatMode,  // Auto, Json, Yaml
}

struct DirectoryRules {
    hidden: Vec<String>,    // never shown
    collapse: Vec<String>,  // shown as "dir/ (N files)"
    expand: Vec<String>,    // always show contents
}
// Merge semantics: language-specific hidden/collapse lists are UNIONED
// with the global lists (additive, never replacing).

struct LanguageConfig {
    detect: Vec<String>,              // marker files
    extensions: Vec<String>,          // file extensions
    parser: ParserKind,               // syn | rustpython | tree-sitter
    hidden: Vec<String>,              // language-specific hidden dirs
    collapse: Vec<String>,            // language-specific collapse dirs
    kinds: HashMap<String, Vec<String>>, // kind → glob patterns
    // Kind classification priority: test > ci > config > build > docs > source
    // A file matches the first kind whose glob matches (highest priority first).
    // This prevents "**/*.py" in source from swallowing test files.
}
```

#### Output Structures

```rust
// Skeleton output
struct SkeletonOutput {
    path: PathBuf,
    languages: Vec<String>,
    tree: Vec<TreeEntry>,
    lines: usize,
}

// Serialized manually to produce clean YAML/JSON tree notation
// (not tagged enums — output uses indented list format)
enum TreeEntry {
    File(String),
    Directory {
        name: String,
        children: Vec<TreeEntry>,
    },
    Collapsed {
        name: String,
        file_count: usize,  // rendered as "name/ (N files)"
    },
}

// Outline output
struct OutlineOutput {
    files: BTreeMap<String, Vec<Symbol>>,
    lines: usize,
}

struct Symbol {
    signature: String,
    line: usize,
    kind: SymbolKind,
    children: Option<Vec<Symbol>>,  // methods, fields under a class/struct
    // In output, children are grouped by kind (fields first, then methods)
    // for readable YAML/JSON structure
}

enum SymbolKind {
    Function,
    Method,
    Class,
    Struct,
    Enum,
    Trait,
    Interface,
    Constant,
    TypeAlias,
    Field,
}
```

### API Design (CLI Interface)

```
xray [OPTIONS] [PATH]

ARGUMENTS:
  [PATH]               Target directory (default: CWD)

LAYERS (subcommands — if omitted, defaults to skeleton):
  skeleton             Smart directory tree — key files shown, noise collapsed
  outline              Function/class/struct signatures with line numbers

FILTER OPTIONS:
  -k, --kind <KIND>    Filter by file kind: source, config, test, ci, docs, build
                       (repeatable, e.g., -k source -k config)
                       skeleton: only shows files/dirs relevant to that kind
                       outline: only parses files of that kind
  -l, --lang <LANG>    Filter by language: rust, python, typescript
                       (repeatable, auto-detected if omitted)
                       If no language detected, skeleton works (no parsing needed);
                       outline skips unparseable files with stderr warning
  --pattern <GLOB>     Scope to files/dirs matching glob (repeatable)
                       skeleton: limits tree to matching subtrees
                       outline: limits parsing to matching files
  --exclude <GLOB>     Exclude files matching glob (repeatable)
  --public             Public symbols only (outline layer)
  --private            Private symbols only (outline layer)

BUDGET:
  -b, --budget <N>     Maximum output lines (0 = unlimited, default from config)

OUTPUT:
  -f, --format <FMT>   Output format: json, yaml, auto (default: auto)
                        auto = YAML on TTY, JSON when piped

GLOBAL:
  -c, --config <FILE>  Config file override
  -V, --version        Print version
  -h, --help           Print help
```

#### Exit Codes

| Code | Meaning |
|------|---------|
| 0 | Success, output within budget |
| 1 | Budget exceeded — stdout is empty, stderr has overage details and hint |
| 2 | Error (invalid args, parse failure, config error, etc.) |

### Implementation Plan

#### Phase 1: Foundation (skeleton layer)

1. Replace scaffold CLI/config with xray-specific structures
2. Implement config loading: embedded defaults → `~/.config/xray/xray.yml` → `--config` override
3. Implement file walking with `ignore` crate
4. Implement directory classification (hidden/collapse/expand) from config
5. Implement smart tree building with collapsing
6. Implement TTY detection + YAML/JSON output
7. Implement budget enforcement with overage error
8. Add `lines:` footer to all output

#### Phase 2: Language detection + file classification

1. Implement language auto-detection from marker files
2. Implement file-to-kind classification using language config globs
3. Wire `--kind` and `--lang` filters into skeleton layer
4. Merge language-specific hidden/collapse lists with global lists

#### Phase 3: Outline layer (Rust first)

1. Define `LanguageParser` trait with `extract_symbols(source: &str) -> Vec<Symbol>`
2. Implement Rust parser using `syn` crate with `Visit` trait
3. Extract: functions, structs (with fields), enums (with variants), traits, impl blocks (methods), constants, type aliases
4. Wire `--public`/`--private` visibility filtering
5. Parallel parsing with `rayon`

#### Phase 4: Outline layer (Python + TypeScript)

1. Implement Python parser using `rustpython-parser` (port patterns from `pyr`)
2. Implement TypeScript/JavaScript parser using `tree-sitter`
3. Handle multi-language repos (detect all, parse each with appropriate parser)

#### Phase 5: Companion Claude skill

1. Create skill in `tatari-skills` that wraps xray invocations
2. Skill instructs the agent on progressive exploration strategy
3. Skill tracks cumulative context budget

## Alternatives Considered

### Alternative 1: Extend pyr to support multiple languages

- **Description:** Add Rust and TypeScript parsers to existing pyr codebase, add budget system
- **Pros:** Less work, proven architecture, existing tests
- **Cons:** pyr's name implies Python-only; its architecture is tightly coupled to `rustpython-parser` subcommand structure (`pyr function`, `pyr class`); no concept of skeleton/tree view; would require major restructuring
- **Why not chosen:** The layered approach (skeleton → outline) is fundamentally different from pyr's per-symbol-type subcommands. Better to start fresh with the right abstraction.

### Alternative 2: Use tree-sitter for all languages

- **Description:** Standardize on tree-sitter for Rust, Python, and TypeScript parsing
- **Pros:** Single parsing framework, consistent API, error-tolerant, language-agnostic queries
- **Cons:** tree-sitter queries are less precise than language-specific parsers for Rust (`syn` gives fully typed AST) and Python (`rustpython-parser` is pure Rust, proven in pyr). Tree-sitter requires C dependencies for grammar compilation.
- **Why not chosen:** Use the best tool per language. `syn` for Rust and `rustpython-parser` for Python are proven, pure-Rust, and more precise. Tree-sitter is the right choice for JS/TS where no pure-Rust alternative exists.

### Alternative 3: Build as a library/MCP server instead of CLI

- **Description:** Expose xray as an MCP server that agents call via tool use, or a Rust library linked into agent frameworks
- **Pros:** Tighter integration, could maintain state between calls (caching)
- **Cons:** CLI is universally accessible (any agent framework, any shell), simpler to test and debug, can be wrapped by MCP or skill later. MCP server adds operational complexity.
- **Why not chosen:** Start as CLI for maximum portability. A skill wraps it for Claude Code. MCP wrapper can be added later if warranted.

### Alternative 4: Token-based budgeting instead of line-based

- **Description:** Count approximate tokens (using tiktoken or similar) instead of lines
- **Pros:** More accurately reflects context window consumption; a 200-char line costs more than a 20-char line
- **Cons:** Token counting adds a dependency and complexity; approximate at best (different models tokenize differently); lines are simpler to reason about and good enough since most output lines are short structured data
- **Why not chosen:** Start with lines. The design doc notes this as a future exploration if lines prove too coarse. Lines are simple, predictable, and zero-dependency.

## Technical Considerations

### Dependencies

**Internal:**
- Patterns from `pyr`: parallel parsing, BTreeMap ordering, YAML/JSON output, TTY detection, cascading pattern match
- Patterns from `lint-unused`: `syn` visitor, span-to-line mapping, context stack

**External crates:**

| Crate | Version | Purpose |
|-------|---------|---------|
| `clap` | 4 | CLI argument parsing (derive) |
| `eyre` | 0.6 | Error handling with context |
| `serde` | 1 | Serialization framework |
| `serde_json` | 1 | JSON output |
| `serde_yaml` | 0.9 | YAML output (deprecated upstream but stable; evaluate `serde_yml` if issues arise) |
| `ignore` | 0.4 | File walking with `.gitignore` respect (includes walkdir + globset) |
| `syn` | 2 | Rust AST parsing (features: full, visit) |
| `rustpython-parser` | 0.4 | Python AST parsing |
| `tree-sitter` | 0.24 | JS/TS parsing framework |
| `tree-sitter-javascript` | 0.23 | JavaScript grammar |
| `tree-sitter-typescript` | 0.23 | TypeScript grammar |
| `rayon` | 1 | Parallel iteration |
| `dirs` | 6 | XDG directory resolution |

### Performance

- **File walking:** `ignore` crate is fast and respects `.gitignore`, avoiding traversal of `node_modules/`, `target/`, etc. Symlinks are not followed (default `ignore` behavior). Empty directories are omitted from output.
- **Parsing:** `rayon` parallelizes file parsing across cores. Files are collected and sorted first for deterministic output (via `BTreeMap`).
- **Skeleton:** No parsing needed — pure filesystem traversal with classification. Should complete in <100ms for typical projects.
- **Outline:** Parsing is the bottleneck. `pyr` parses thousands of Python files in <1s with rayon. Expect similar for Rust (`syn`) and JS/TS (tree-sitter).
- **Budget check:** O(1) — just count serialized output lines before printing.
- **Binary files:** Outline skips non-UTF-8 files silently (no error, no output for that file).
- **No config file:** First run with no `~/.config/xray/xray.yml` works using embedded defaults. Config is optional.

### Security

- xray is a read-only tool — it never modifies files
- File paths in output are relative to the target directory (no leaking of absolute paths beyond the root)
- Config file is loaded from trusted locations only (`~/.config/xray/xray.yml` or explicit `--config`)
- No network access, no execution of discovered code

### Testing Strategy

- **Fixture-based:** `tests/fixtures/` contains sample projects per language (Rust, Python, TypeScript) with known structures
- **Unit tests per module:**
  - Config loading (defaults, merge, override)
  - Directory classification (hidden, collapse, expand)
  - Skeleton tree building (collapsing behavior, file ordering)
  - Language detection (single, multi, override)
  - Each language parser (function/class/struct extraction, visibility filtering)
  - Budget enforcement (within, exceeded, unlimited)
  - Output formatting (YAML, JSON, TTY detection)
- **Integration tests:** End-to-end CLI invocation against fixture projects, checking exact output
- **CI:** `.otto.yml` with lint, check, test, coverage tasks (already scaffolded)
- **Coverage target:** 80%+ line coverage via `cargo llvm-cov`

### Rollout Plan

1. **v0.1** — Skeleton layer with smart collapsing, budget enforcement, config system, language detection. No parsing.
2. **v0.2** — Outline layer for Rust (syn-based). `--kind` and `--pattern` filtering.
3. **v0.3** — Outline layer for Python (rustpython-parser) and TypeScript (tree-sitter).
4. **v0.4** — Companion Claude skill in tatari-skills.
5. **v1.0** — Stabilized after real-world agent usage and feedback.

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| tree-sitter C deps cause build issues | Medium | Medium | Isolate behind feature flag; JS/TS support is Phase 4 |
| Line-based budget too coarse for agent needs | Low | Medium | Design allows swapping to token-based later; `lines:` footer lets agents self-regulate |
| Config becomes complex / hard to maintain | Low | Low | Ship sensible embedded defaults; config is optional override |
| Language parsers lag behind language spec changes | Low | Medium | syn tracks Rust editions; rustpython-parser is actively maintained; tree-sitter grammars are community-maintained |
| Skeleton collapsing hides files agents actually need | Medium | Low | Agent can always use `--pattern` to force expansion of collapsed dirs; collapse rules are config-tunable |
| Multi-language repos produce confusing mixed output | Low | Medium | `--lang` filter lets agent focus on one language at a time |
| Very large monorepos (50K+ files) slow skeleton | Low | Low | `ignore` crate is fast; hidden dirs skip traversal entirely; collapse avoids enumeration |
| Symlinked directories cause duplicate entries | Low | Low | `ignore` crate does not follow symlinks by default |

## Open Questions

- [ ] Should `skeleton` show file sizes (bytes) next to filenames? Useful signal for agents to know if a file is 10 lines or 1000 lines before reading it.
- [ ] Should collapsed directories also show dominant file types? e.g., `data/ (50 files: 48 .csv, 2 .json)` — useful but adds output lines.
- [ ] For `outline`, should we support a `--depth` flag to control nesting? e.g., `--depth 1` shows only top-level symbols, `--depth 2` includes methods/fields.
- [ ] Should the skill maintain state across invocations (e.g., remembering what the agent already explored) or remain stateless?
- [ ] For the config, should we support per-project `.xray.yml` overrides (checked into repos) in addition to the global config?

## Agent Workflow Example

An agent receives a task: "Fix the authentication bug in the user service."

```bash
# Step 1: Orient (cost: ~25 lines)
$ xray skeleton
# Agent sees: src/api/, src/models/, src/auth/, tests/

# Step 2: Drill into auth (cost: ~15 lines)
$ xray outline --pattern "src/auth/**"
# Agent sees: verify_token(), create_session(), AuthMiddleware struct

# Step 3: Read specific file (agent uses Read tool directly)
# Agent now knows exactly which file and line to look at
```

Total context spent on exploration: ~40 lines instead of reading 5+ full files blind.

**Multi-language repo:**

```bash
# Skeleton detects both languages
$ xray skeleton
# languages: [python, rust]
# tree shows both pyproject.toml and Cargo.toml, src/ and myapp/

# Agent focuses on the Python side for its task
$ xray outline --lang python --kind source --budget 80
```

## Differences from pyr

| Dimension | pyr | xray |
|-----------|-----|------|
| Languages | Python only | Rust, Python, JS/TS |
| Budget | None | `--budget` with overage reporting |
| Tree view | None | Smart skeleton with collapsing |
| File classification | None | `--kind` (source, config, test, ci, docs, build) |
| Low-signal dirs | Hardcoded ignores | Config-driven collapse vs hide vs expand |
| Output footer | None | Always shows `lines:` count |
| Config | None | `~/.config/xray/xray.yml` with language profiles |
| Subcommands | Per symbol type | Layered (skeleton, outline) |
