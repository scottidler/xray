# Design Document: xray v0.2 - Compact Output, Truncation, and Dotfile Hiding

**Author:** Scott Idler + Claude
**Date:** 2026-03-21
**Status:** Implemented
**Review Passes Completed:** 5/5

## Summary

xray v0.1 shipped with skeleton and outline layers for Rust, Python, and TypeScript. Real-world testing on repos from 2k to 66k lines exposed three friction points: budget exceeded produces empty output (wasted round trips), dotfiles clutter the skeleton, and outline JSON/YAML is too verbose for agent scanning. This doc proposes four targeted improvements for v0.2.

## Problem Statement

### Background

xray v0.1 was deployed and tested against three repos of increasing size:

| Repo | Lines (Rust) | Skeleton | Outline (source, public) |
|------|-------------|----------|--------------------------|
| xray | 2.5k | 30 lines, clean | fits easily |
| second-brain | 21k | 117 lines, clean | 1618 lines, required scoping |
| loopr | 66k | 64KB JSON, noisy | 50-60KB per module even with --public |

### Problem

**1. Budget exceeded = zero output.** When outline output exceeds `--budget`, xray exits with code 1 and empty stdout. The agent gets nothing - all parsing work is discarded. In testing, this caused 3 consecutive wasted round trips as the agent guessed budgets: `--budget 150` (1618 lines), `--budget 200` (568 lines), `--budget 300` (317 lines). Each attempt re-parsed every file and returned nothing useful.

**2. Dotfiles pollute the skeleton.** On loopr, the skeleton was 64KB because `.worktrees/` (git worktree artifacts with duplicated repo contents), `.taskstore/` (task tracking data), and `.claude/` (settings) were fully enumerated. These are never useful for codebase understanding. The config-driven hidden list catches specific patterns like `.git` and `node_modules`, but misses any dotfile not explicitly listed.

**3. Outline output is too verbose.** A single symbol in JSON is 5-8 lines (signature, line, kind, visibility, children). An agent scanning for "where does auth live?" wants `src/auth.rs:45 pub fn verify_token(...)`, not a nested data structure. The structured format is correct for machine consumption, but agents reading output as text need a compact alternative.

### Goals

- Budget enforcement should always return useful partial output, never empty stdout
- Dotfiles should be hidden by default with an opt-in flag to show them
- Outline should offer a compact one-line-per-symbol format for agent scanning
- All changes should be backward-compatible with existing config and CLI usage

### Non-Goals

- Token-based budgeting (line-based is sufficient for v0.2)
- Compact mode for skeleton (tree structure doesn't reduce to one-line-per-entry cleanly)
- Interactive/incremental exploration (stateless CLI model is fine)
- MCP server wrapper (out of scope, CLI-first)

## Proposed Solution

### Overview

Four changes, all additive:

1. **`--hidden` flag** - skip dotfiles by default, opt-in to show them
2. **Tighter hidden defaults** - add `.worktrees`, `.taskstore`, `.claude`, `.env`, `.env.*` to config
3. **Truncate instead of fail** - budget exceeded emits first N lines with truncation footer
4. **`--compact` / `-1` flag** - one-line-per-symbol outline output

### Change 1: Skip dotfiles by default

Add a `--hidden` CLI flag (global, default false). When passed, dotfiles are *included*. When omitted (default), any file or directory starting with `.` is skipped before the config-driven hidden check.

**CLI:**
```
--hidden    Show hidden files and directories (dotfiles)
```

**Skeleton walker** (`skeleton.rs::build_tree`): Add check at top of the entry loop, before the config-driven hidden pattern match:
```rust
if !ctx.show_hidden && name.starts_with('.') {
    continue;
}
```

**Outline collector** (`outline/mod.rs::collect_files_recursive`): Same dotfile check for both directory and file names.

**Threading:** Add `show_hidden: bool` to `TreeContext` and thread through `build_skeleton()`, `build_outline()`, `collect_source_files()`, `collect_files_recursive()`.

**Interaction with config hidden list:** When `--hidden` is passed, dotfiles pass the dotfile check but still go through the config-driven hidden list. So `.git` stays hidden even with `--hidden` because it's in `directories.hidden`. This is correct behavior - `--hidden` means "show dotfiles that aren't explicitly hidden", not "show everything".

### Change 2: Tighter hidden defaults

Add to `directories.hidden` in `xray.yml`:

```yaml
- .worktrees
- .taskstore
- .claude
- .env
- ".env.*"
```

These serve as belt-and-suspenders for `--hidden` mode. Under default behavior (dotfiles hidden), they're redundant. Under `--hidden`, they prevent the most common noise directories from appearing.

### Change 3: Truncate instead of fail on budget exceeded

Replace the `check_budget` / `BudgetExceeded` error pattern with truncation.

**New types in `output.rs`:**
```rust
pub struct TruncationInfo {
    pub truncated: bool,
    pub shown: usize,
    pub total: usize,
}
```

**New function:**
```rust
pub fn truncate_to_budget(output: &str, budget: usize) -> (String, TruncationInfo)
```

Logic:
- If budget is 0 (unlimited), return output as-is with `truncated: false`
- Otherwise, count lines in the serialized output string
- If within budget, return as-is with `truncated: false`
- If over budget, take the first `budget` lines and return with `truncated: true`, `shown: budget`, `total: actual_line_count`

**Truncation footer** appended when truncated:
- YAML mode: `truncated: true\nshown: 150\ntotal: 1618\n`
- JSON mode: `{"truncated": true, "shown": 150, "total": 1618}\n`
- Compact mode: `...truncated 150/1618 lines shown\n`

**Example: compact + truncation combined** (`xray outline -1 --budget 5`):
```
src/main.rs:22 pub fn main() -> Result < () >
src/config.rs:8 pub struct Config
src/config.rs:9   pub budget: usize
src/config.rs:10   pub format: FormatMode
src/config.rs:20 pub fn load(path: Option < & Path >) -> Result < Config >
...truncated 5/42 lines shown
```

**Exit code:** Always 0. Remove `process::exit(1)` for budget. Exit 2 remains for actual errors (bad args, missing directory).

**main.rs flow change:**
```
Before: build -> count lines -> check_budget (fail if over) -> serialize -> print
After:  build -> serialize -> truncate_to_budget -> print (always succeeds)
```

This eliminates the line-counting functions (`count_output_lines`) as the primary budget mechanism. Line counting is now done on the serialized string, which eliminates any mismatch between estimated and actual line counts.

**Important: truncation produces invalid YAML/JSON.** Cutting a serialized YAML or JSON document mid-way will produce a syntactically invalid fragment. This is acceptable because:
1. The primary consumer is an agent reading text, not a JSON/YAML parser
2. The truncation footer clearly signals the output is incomplete
3. For machine consumption, agents should narrow scope to avoid truncation rather than parsing truncated output
4. Compact mode (`-1`) is plain text and truncates cleanly

### Change 4: Compact output mode

Add `--compact` / `-1` CLI flag (global). Only meaningful for outline; ignored for skeleton.

**Format:** One line per symbol, children indented under parents:
```
src/main.rs:22 pub fn main() -> Result < () >
src/config.rs:8 pub struct Config
src/config.rs:9   pub budget: usize
src/config.rs:10   pub format: FormatMode
src/config.rs:20 pub fn load(path: Option < & Path >) -> Result < Config >
src/outline/mod.rs:56 pub trait LanguageParser
src/outline/mod.rs:57   fn extract_symbols(& self, source: & str) -> Result < Vec < Symbol > >
```

Format per line: `{file_path}:{line} {visibility_prefix}{signature}`

- The `signature` field comes directly from the parser (e.g. `fn main() -> Result < () >`, `struct Config`). Parsers already include the keyword (`fn`, `struct`, `enum`, `trait`, etc.) in the signature.
- Visibility prefix: `pub ` for Public, empty for Private
- Children indented with 2 spaces to show nesting (methods under impl, fields under struct, variants under enum)
- Footer: `lines: N`

**Implementation in `output.rs`:**
```rust
pub fn serialize_compact(outline: &OutlineOutput) -> String
```

Iterates `outline.files` (already sorted by BTreeMap), for each file iterates symbols recursively, writes one line per symbol.

**In `main.rs`:** When `cli.compact` is true in the outline branch, call `serialize_compact` instead of `serialize`. The result is a plain string that goes through the same `truncate_to_budget` path.

**Interaction with `--format`:** When `--compact` is active, `--format` is ignored. Compact output is always plain text. No warning needed - compact is a more specific request that supersedes format selection.

### Data Model

No new data structures for domain types. `Symbol`, `OutlineOutput`, `SkeletonOutput` are unchanged. New types are limited to `TruncationInfo` in `output.rs`.

### API Design (CLI)

New flags added to the global argument group:

| Flag | Short | Type | Default | Description |
|------|-------|------|---------|-------------|
| `--hidden` | - | bool | false | Include dotfiles/dotdirs in output |
| `--compact` | `-1` | bool | false | One-line-per-symbol outline output |

Existing flags unchanged. Exit codes simplified:

| Code | Meaning |
|------|---------|
| 0 | Success (may include truncation) |
| 2 | Error (bad args, not a directory, config error) |

Exit code 1 is removed (was budget exceeded).

### Implementation Plan

#### Phase 1: CLI and config
1. Add `--hidden` and `--compact` flags to `cli.rs`
2. Add new hidden entries to `xray.yml`

#### Phase 2: Dotfile hiding
1. Add `show_hidden` to `TreeContext` in `skeleton.rs`
2. Add dotfile check in `build_tree()` loop
3. Thread `show_hidden` through `build_skeleton()` signature
4. Add dotfile check in `collect_files_recursive()` in `outline/mod.rs`
5. Thread `show_hidden` through `build_outline()`, `collect_source_files()` signatures

#### Phase 3: Budget truncation
1. Remove `BudgetExceeded` and `check_budget` from `output.rs`
2. Add `TruncationInfo` and `truncate_to_budget` to `output.rs`
3. Update `main.rs` to use truncation instead of fail-on-exceed

#### Phase 4: Compact output
1. Add `serialize_compact()` to `output.rs`
2. Add compact branch in `main.rs` outline handler
3. Update tests

## Alternatives Considered

### Alternative 1: Token-based budgeting instead of line-based truncation
- **Description:** Count approximate tokens per line and truncate at a token budget rather than line budget
- **Pros:** More accurately reflects context window consumption
- **Cons:** Adds tiktoken or similar dependency, complicates the mental model, line-based is good enough in practice
- **Why not chosen:** Premature optimization. Line counts are simple, predictable, and correlate well enough with token consumption. Can revisit if agents report token pressure despite line budgets.

### Alternative 2: Progressive/streaming output instead of truncation
- **Description:** Stream output line-by-line and stop at budget, rather than building full output then truncating
- **Pros:** More memory-efficient for huge repos, no wasted work
- **Cons:** Requires rearchitecting serialization, rayon parallel parsing doesn't lend itself to streaming, more complex implementation
- **Why not chosen:** Truncation on the serialized string is simple and correct. The full parse is fast (rayon parallelism). The bottleneck is context window consumption, not CLI runtime.

### Alternative 3: `--no-dotfiles` instead of `--hidden`
- **Description:** Default to showing dotfiles, add flag to hide them
- **Pros:** More backward-compatible with v0.1 behavior
- **Cons:** The v0.1 behavior was wrong for agents. Every test showed dotfile noise. Making the bad behavior opt-out means most users get worse results by default.
- **Why not chosen:** The whole point is to fix the default experience. `--hidden` to opt-in matches `ls` convention (hidden by default, `-a` to show).

### Alternative 4: Compact as a `--format` option instead of separate flag
- **Description:** Add `--format compact` alongside `json`/`yaml`/`auto`
- **Pros:** Fewer flags, unified format concept
- **Cons:** Compact is only meaningful for outline, not skeleton. Mixing it into format creates confusion about what `--format compact` means for skeleton. Also, compact is orthogonal to the structured formats - you might want compact YAML in the future.
- **Why not chosen:** `--compact` as a separate boolean flag is cleaner. It says "flatten the output" independently of the serialization format.

## Technical Considerations

### Dependencies

No new dependencies. All changes use existing crates (clap, serde, serde_yaml, serde_json).

### Performance

- Truncation adds negligible overhead (one pass over a string to count/split lines)
- Dotfile check (`name.starts_with('.')`) is O(1) per entry, no measurable impact
- Compact serialization is a simple string-building loop, faster than serde serialization

### Testing Strategy

- Update existing budget tests to verify truncation behavior (partial output, truncation footer, exit 0)
- Add tests for dotfile hiding: temp dir with dotfiles, verify they're excluded/included with flag
- Add tests for compact output: known symbols in, verify one-line-per-symbol format out
- Integration verification: run against xray, second-brain, and loopr repos

### Rollout Plan

1. **v0.2.0** - all four changes ship together (they're independent but form a coherent "agent experience" improvement)
2. Update the `/xray` Claude skill to reflect new flags and behavior
3. Test with real agent workflows on second-brain and loopr

## Risks and Mitigations

| Risk | Likelihood | Impact | Mitigation |
|------|------------|--------|------------|
| `--hidden` default change breaks workflows relying on dotfile visibility | Low | Low | `--hidden` flag provides opt-in. Note: `.github/` CI files require `--hidden` to be visible even with `--kind ci`, since the dotfile check precedes classification |
| Truncated output misleads agent (partial data looks complete) | Medium | Medium | Clear truncation footer with shown/total counts; agent can narrow scope |
| Compact format loses structure (no kind/visibility metadata) | Low | Low | Visibility shown as `pub` prefix; kind is implicit in signature. Structured YAML/JSON remains available |
| Removing exit code 1 breaks scripts checking for budget exceeded | Low | Low | No known consumers of exit code 1; documented behavior change |

## Open Questions

- [x] Should compact mode indent children (2 spaces for methods under struct/impl) or fully flatten everything? **Decision: indent children with 2 spaces.** Flat output loses the struct-method relationship which is valuable context for agents.
- [x] Should `--hidden` also apply to the outline's default "skip non-source/test files" behavior, or only to the dotfile check? **Decision: dotfile check only.** The kind-based filtering (`--kind`) is a separate concern. `--hidden` controls filesystem-level visibility, `--kind` controls semantic filtering. They compose independently.
- [x] Should truncation preserve the `lines:` footer from the data structure, or let the truncation footer replace it? **Decision: truncation footer replaces it.** The `lines:` field in the data structure reflects the un-truncated total; the truncation footer shows both shown and total, which is more informative.
